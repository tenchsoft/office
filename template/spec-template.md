# Spec Template

`plans/spec/<product>/<id>.md`에 들어갈 문서의 골격.

**원칙: 코드 참고 없이 사용자 관점에서만 작성.** 파일 경로/함수명/내부 상태 명칭 금지.
spec은 "사용자가 무엇을 보고, 무엇을 하고, 무엇이 일어나는가"의 계약. 구현 방식이
바뀌어도 spec은 그대로 유효해야 한다.

`<product>`는 `apps/<product>/`의 디렉토리명 (`docs`, `kodocs`, `sheets`, `slides`, `research`, `study`).
`<id>`는 spec/implement/test 세 폴더에서 동일하게 사용 (kebab-case 슬러그).

---

## 골격

```markdown
# Spec: <id>

## 한 줄 정의
사용자가 <어디서> <어떻게> 하면 <무엇이> 일어난다.

## 진입점
이 기능을 발동하는 모든 방법.
- 메뉴: <상위> → <항목>
- 단축키: <key combo>
- 툴바: <버튼 이름>
- 그 외: 우클릭 / 드래그 / 자동 (시간 기반) / ...

## 사용자 흐름
정상 시나리오를 시간 순서대로. 각 단계에 **사용자가 보는 시각적 결과**까지.
1. 사용자가 ...
2. 시스템이 ... (사용자가 보는 변화: ...)
3. ...

## 성공 조건 (Acceptance Criteria)
사용자가 직접 관찰 가능한 조건만. 내부 상태 언급 금지.
- [ ] 사용자가 <행동>하면 <결과>가 화면에 보인다.
- [ ] ...

## 실패 / 취소 흐름
정상 외 경로 모두. 각각 사용자가 보는 결과 명시.
- 사용자가 <취소>하면 → 화면이 ... (변화 없음 / 원래대로)
- 입력이 잘못되면 → <에러 표시 형태>
- 자원이 없으면 (파일 없음 등) → <사용자 메시지>
- 다른 모달이 활성 중이면 → <라우팅 정책>

## 경계 / 예외
- 라운드트립: 같은 행동을 두 번/연속으로 → ...
- 경계값: 빈 입력 / 최대 / 0 → ...
- 동시성: 다른 작업 진행 중에 → ...
- 다중 인스턴스: 여러 탭/세션에서 → ...

## 범위 외
이 spec이 다루지 않는 것 (별도 spec으로 분리되었거나 후속 작업).
- ...
```

---

## 작성 규칙

- **금지**: 파일 경로 (`apps/...`), 함수명 (`handle_pointer_down`), 변수/필드명 (`state.show_comments`), 라이브러리 이름, 코드 스니펫.
- **허용**: 사용자가 보는 시각적 요소 (메뉴 항목 이름, 버튼 라벨, 토스트 문구, 다이얼로그 제목).
- **단위**: 한 spec = 한 사용자 액션. "Find 기능 전체"가 아니라 "Find 모달 열기" / "Aa 토글 켜기" / "다음 일치로 이동"을 각각.
- **acceptance criteria**: 모두 검증 가능해야 한다. "잘 동작한다", "자연스럽다" 같은 추상 표현 금지. 한 항목씩 테스트로 매핑되어야 함.
- **길이**: 50–100줄 권장. spec이 200줄 넘으면 단위가 너무 큼 — 둘로 쪼개라.

## 예시 (id: file-menu-open)

```markdown
# Spec: file-menu-open

## 한 줄 정의
사용자가 File → Open 메뉴 항목으로 .docx 파일을 새 탭에 연다.

## 진입점
- 메뉴: File → Open
- 단축키: Ctrl+O (모달 비활성 시에만)

## 사용자 흐름
1. 사용자가 File 메뉴를 연다 → 메뉴가 펼쳐지고 Open 항목이 활성 상태.
2. 사용자가 Open을 클릭 → 메뉴가 즉시 닫힌다.
3. native 파일 다이얼로그가 뜬다.
4. 사용자가 .docx 파일을 선택하고 OK → 다이얼로그가 닫힌다.
5. 새 탭이 추가되고 활성화된다 (탭 라벨에 파일명 표시).
6. 본문 영역에 첫 페이지가 렌더링된다.
7. "Opened: <filename>" 토스트가 표시되고 3초 후 자동 사라진다.
8. 저장 상태 표시는 "saved"로 시작.

## 성공 조건
- [ ] File → Open 클릭 시 native 다이얼로그가 표시된다.
- [ ] 유효한 .docx 선택 시 새 탭이 추가되고 자동 활성화된다.
- [ ] 새 탭의 저장 상태가 "saved"로 시작한다.
- [ ] 본문에 첫 페이지가 보인다.
- [ ] 성공 토스트가 표시되고 3초 후 사라진다.

## 실패 / 취소 흐름
- 사용자가 다이얼로그 Cancel → 화면 변화 없음, 토스트 없음.
- 지원하지 않는 형식 (.txt, .pdf 등) 선택 → "Unsupported format" 에러 토스트, 새 탭 미생성.
- 파일 없음 / 권한 없음 → "Cannot open: <reason>" 에러 토스트, 새 탭 미생성.
- 손상된 .docx → "Failed to parse" 에러 토스트, 새 탭 미생성.
- 모달 활성 중 Ctrl+O → 무시 (다이얼로그 안 뜸).

## 경계 / 예외
- 같은 파일을 다시 Open → 새 탭이 또 생긴다 (중복 허용).
- 활성 탭에 미저장 변경이 있어도 Open은 진행 (활성 탭 보존).
- 빈 .docx (0 페이지) → 빈 본문으로 탭이 열린다.

## 범위 외
- 비-.docx 형식 import (별도 spec).
- "기존 탭으로 포커스" 정책 (현재는 항상 새 탭).
- 최근 파일 목록 관리 정책 (별도 spec).
```
